# 软件设计说明书

## 1. 引言

### 1.1 编写目的

本文档用于描述学校部门财务管理系统的总体架构、模块划分、数据库设计、接口设计、页面设计以及关键约束，为系统实现、测试和答辩提供依据。

### 1.2 设计目标

- 保证系统结构清晰，易于课程答辩说明。
- 保证数据库、接口和页面三层一致。
- 保证核心业务在 4 周周期内可落地。
- 为后续扩展预算分析、导出与审计功能预留空间。

## 2. 总体设计

### 2.1 架构方案

系统采用前后端分离架构。

- 前端：Vue 3 + Vite + Element Plus
- 后端：Spring Boot 3 + MyBatis-Plus
- 数据库：MySQL 8
- 接口风格：REST

### 2.2 分层设计

| 层次 | 说明 |
| --- | --- |
| 表示层 | 前端页面、表单、图表、菜单、路由 |
| 接口层 | 控制器接收请求并返回统一响应 |
| 业务层 | 登录校验、主数据校验、统计口径处理 |
| 数据访问层 | 通过 MyBatis-Plus 或 JDBC 访问数据库 |
| 数据层 | 存储用户、部门、账户、类别、收支、备份数据 |

### 2.3 系统模块

1. 登录与权限管理模块
2. 基础信息管理模块
3. 收入信息管理模块
4. 支出信息管理模块
5. 财务统计与报表模块
6. 数据备份与恢复模块

### 2.4 技术原则

- 核心主数据采用“停用优先”策略，避免误删。
- 报表统一读取有效状态业务记录。
- 页面、接口、表结构尽量保持同名映射。
- 课程设计版本优先保证主流程完整、口径一致和演示稳定。

## 3. 模块设计

### 3.1 登录与权限管理模块

#### 功能职责

- 用户登录与退出
- token 下发
- 角色识别
- 菜单与按钮权限控制

#### 主要对象

- `sys_user`
- `sys_role`
- `sys_user_role`

### 3.2 基础信息管理模块

#### 功能职责

- 部门管理
- 账户管理
- 收入类别管理
- 支出类别管理
- 用户管理

#### 设计说明

- 主数据保存唯一性和状态字段。
- 若被收支记录引用，则禁止物理删除。

### 3.3 收入信息管理模块

#### 功能职责

- 收入记录新增、修改、停用、查询

#### 核心校验

- 金额大于 0
- 部门、账户、类别有效
- 发生日期不为空

### 3.4 支出信息管理模块

#### 功能职责

- 支出记录新增、修改、停用、查询

#### 核心校验

与收入模块一致。

### 3.5 财务统计与报表模块

#### 功能职责

- 周统计
- 月统计
- 年统计
- 自定义区间统计
- 首页仪表盘统计

#### 输出结构

- 查询条件
- 收入总额
- 支出总额
- 结余
- 按类别汇总
- 按部门汇总
- 按时间趋势汇总

### 3.6 数据备份与恢复模块

#### 功能职责

- 手动创建数据库逻辑备份
- 备份记录管理
- 备份文件下载
- 恢复确认与执行

#### 设计限制

- 不实现自动定时备份
- 不实现分布式存储

## 4. 数据库设计

### 4.1 表清单

| 表名 | 说明 |
| --- | --- |
| `sys_user` | 用户表 |
| `sys_role` | 角色表 |
| `sys_user_role` | 用户角色关联表 |
| `dept_info` | 部门表 |
| `account_info` | 账户表 |
| `income_category` | 收入类别表 |
| `expense_category` | 支出类别表 |
| `income_record` | 收入记录表 |
| `expense_record` | 支出记录表 |
| `backup_record` | 备份记录表 |

### 4.2 关键表设计

#### 4.2.1 `sys_user`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| username | varchar(50) | 登录名，唯一 |
| password | varchar(100) | 密码 |
| real_name | varchar(50) | 真实姓名 |
| phone | varchar(20) | 联系电话 |
| status | tinyint | 1 启用，0 停用 |
| created_at | datetime | 创建时间 |

#### 4.2.2 `dept_info`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| dept_code | varchar(30) | 部门编号 |
| dept_name | varchar(100) | 部门名称 |
| leader_name | varchar(50) | 负责人 |
| phone | varchar(20) | 联系电话 |
| status | tinyint | 状态 |
| created_at | datetime | 创建时间 |

#### 4.2.3 `account_info`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| account_code | varchar(30) | 账户编号 |
| account_name | varchar(100) | 账户名称 |
| owner_unit | varchar(100) | 开户单位 |
| description | varchar(255) | 说明 |
| status | tinyint | 状态 |
| created_at | datetime | 创建时间 |

#### 4.2.4 `income_record`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| record_no | varchar(40) | 收入编号 |
| dept_id | bigint | 部门 ID |
| account_id | bigint | 账户 ID |
| category_id | bigint | 收入类别 ID |
| amount | decimal(12,2) | 金额 |
| occurred_on | date | 发生日期 |
| operator_name | varchar(50) | 经办人 |
| remark | varchar(255) | 备注 |
| status | tinyint | 状态 |
| created_at | datetime | 创建时间 |

#### 4.2.5 `expense_record`

字段与 `income_record` 基本一致，仅 `category_id` 对应支出类别。

#### 4.2.6 `backup_record`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| backup_name | varchar(100) | 备份名称 |
| file_path | varchar(255) | 文件路径 |
| file_size | bigint | 文件大小 |
| created_by | varchar(50) | 操作人 |
| created_at | datetime | 创建时间 |
| restored_at | datetime | 最近恢复时间 |
| remark | varchar(255) | 备注 |

### 4.3 关系说明

- 一个用户可拥有一个角色集合，本项目默认一个用户对应一个主角色。
- 一个部门可关联多条收入和支出记录。
- 一个账户可关联多条收入和支出记录。
- 一个类别可关联多条对应业务记录。

## 5. 接口设计

### 5.1 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

### 5.2 认证接口

#### `POST /auth/login`

请求参数：

```json
{
  "username": "admin",
  "password": "123456"
}
```

响应参数：

```json
{
  "token": "mock-token",
  "username": "admin",
  "realName": "系统管理员",
  "roles": ["ADMIN"]
}
```

#### `POST /auth/logout`

用于用户退出登录。

### 5.3 主数据与用户管理接口

- `GET /users`
- `POST /users`
- `PUT /users/{id}`
- `PATCH /users/{id}/status`
- `GET /departments`
- `POST /departments`
- `PUT /departments/{id}`
- `PATCH /departments/{id}/status`
- `GET /accounts`
- `POST /accounts`
- `PUT /accounts/{id}`
- `PATCH /accounts/{id}/status`
- `GET /income-categories`
- `POST /income-categories`
- `PUT /income-categories/{id}`
- `PATCH /income-categories/{id}/status`
- `GET /expense-categories`
- `POST /expense-categories`
- `PUT /expense-categories/{id}`
- `PATCH /expense-categories/{id}/status`

### 5.4 收支记录接口

- `GET /incomes`
- `POST /incomes`
- `PUT /incomes/{id}`
- `PATCH /incomes/{id}/status`
- `GET /expenses`
- `POST /expenses`
- `PUT /expenses/{id}`
- `PATCH /expenses/{id}/status`

查询条件支持：部门、账户、类别、开始日期、结束日期。

### 5.5 报表接口

- `GET /reports/dashboard`
- `GET /reports/weekly`
- `GET /reports/monthly`
- `GET /reports/yearly`
- `GET /reports/custom`

统一返回：

- `query`
- `totalIncome`
- `totalExpense`
- `balance`
- `categoryStats`
- `departmentStats`
- `trendStats`

### 5.6 备份接口

- `GET /backups`
- `POST /backups`
- `GET /backups/{id}/download`
- `POST /backups/{id}/restore`

### 5.7 二期预留接口

- `GET /reports/export/word`
- `GET /reports/export/pdf`

## 6. 页面设计

### 6.1 页面清单

| 页面 | 说明 |
| --- | --- |
| 登录页 | 用户登录入口 |
| 首页仪表盘 | 展示核心统计摘要和趋势图 |
| 用户管理页 | 管理员维护用户 |
| 部门管理页 | 管理员维护部门 |
| 账户管理页 | 管理员维护账户 |
| 收入类别管理页 | 管理员维护收入类别 |
| 支出类别管理页 | 管理员维护支出类别 |
| 收入记录页 | 收入录入与查询 |
| 支出记录页 | 支出录入与查询 |
| 报表中心页 | 周、月、年、自定义报表 |
| 备份管理页 | 备份创建、查看、恢复 |

### 6.2 页面布局

- 顶部：系统标题、用户信息、退出按钮
- 左侧：菜单导航
- 中间：业务内容区

### 6.3 交互规则

- 列表页支持搜索、状态展示和弹窗维护。
- 表单页统一提供保存和取消功能。
- 停用操作需二次确认。
- 恢复备份需明显警示。

## 7. 安全与异常设计

- 未登录访问接口时返回未授权状态。
- 前端按角色显示菜单，但最终权限以后端为准。
- 删除类操作默认使用停用替代物理删除。
- 报表时间区间非法时返回明确错误提示。
- 备份文件不存在时需要返回可理解的业务异常信息。

## 8. 部署设计与对应关系

### 8.1 部署设计

- 前端部署为静态资源或通过 Vite 开发服务运行。
- 后端部署为 Spring Boot 应用，默认端口为 `8080`。
- 数据库独立部署为 MySQL 实例。
- 备份文件保存在后端服务指定目录。

### 8.2 设计与实现对应关系

- 模块设计对应 `10-源代码/backend/src/main/java/com/classdesign/finance` 中的 `controller`、`service`、`mapper` 分层结构。
- 页面设计对应 `10-源代码/frontend/src/pages` 中的各功能页面。
- 数据表设计对应 `11-数据库/财务管理系统.sql` 中的建表脚本。

## 9. 结论

本设计采用清晰的分层方式和较稳定的技术栈，既满足课程设计“可设计、可实现、可测试、可答辩”的要求，也为后续功能扩展和文档维护保留了足够空间。
