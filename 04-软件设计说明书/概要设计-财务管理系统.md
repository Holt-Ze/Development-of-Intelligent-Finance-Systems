# 概要设计

## 1. 引言

### 1.1 编写目的

本文档用于说明基于 Web 的学校部门财务管理系统在实现层面之前的总体设计方案，明确系统的架构思路、模块划分、数据边界、接口关系和部署方式，为详细设计、编码实现、测试设计和课程答辩提供统一依据。

### 1.2 项目范围

系统面向学校内部部门财务管理场景，支持管理员和财务员两类角色，覆盖登录认证、用户管理、部门管理、账户管理、收入类别管理、支出类别管理、收入记录管理、支出记录管理、报表统计以及数据备份与恢复。

### 1.3 设计依据

- 《软件开发计划书-财务管理系统》
- 《需求规格说明书-财务管理系统》
- 当前仓库中的前后端源码、数据库脚本和系统建模图
- 课程设计文档编写规范

## 2. 设计目标与原则

### 2.1 设计目标

- 在 4 周课程设计周期内形成可运行、可展示、可归档的财务管理系统。
- 保证业务主流程完整，即“登录 -> 主数据维护 -> 收支录入 -> 报表统计 -> 数据备份”可顺畅演示。
- 保证文档、数据库、接口、页面和测试材料之间可以相互追踪。

### 2.2 设计原则

- 采用前后端分离结构，便于模块职责清晰和联调。
- 以角色权限为核心控制系统入口，防止非授权访问。
- 主数据与业务数据分表管理，减少耦合并方便扩展。
- 统计口径统一，报表模块仅汇总有效状态的收支记录。
- 保持实现复杂度适中，优先满足课程设计展示与验收要求。

## 3. 系统总体架构

### 3.1 架构模式

系统采用典型 B/S 架构，浏览器作为客户端访问前端单页应用，前端通过 REST 接口调用后端服务，后端再访问 MySQL 数据库完成持久化和统计查询。

### 3.2 技术架构

| 层次 | 主要技术 | 职责说明 |
| --- | --- | --- |
| 表现层 | Vue 3、Vite、Element Plus、Vue Router | 负责页面展示、交互校验、路由切换和权限菜单控制 |
| 接口通信层 | Axios、HTTP、JSON | 负责前后端数据传输和统一接口调用 |
| 业务层 | Spring Boot 3、Spring Validation | 负责登录认证、业务规则校验、异常处理和服务编排 |
| 数据访问层 | MyBatis-Plus、JdbcTemplate | 负责实体 CRUD、条件查询和报表聚合查询 |
| 数据层 | MySQL 8 | 负责存储用户、角色、主数据、收支记录与备份记录 |

### 3.3 逻辑结构

| 子系统 | 组成模块 | 说明 |
| --- | --- | --- |
| 前端子系统 | 登录页、仪表盘、主数据页面、收支页面、报表页面、备份页面 | 统一通过 `MainLayout` 组织页面框架 |
| 后端子系统 | 控制器、服务、数据访问对象、统一响应与异常处理 | 提供面向页面的 REST 风格接口 |
| 数据子系统 | 10 张核心业务表和初始化数据脚本 | 支撑演示数据、统计分析和备份记录 |

### 3.4 部署结构

| 节点 | 部署内容 | 默认说明 |
| --- | --- | --- |
| 浏览器客户端 | 前端静态资源 | 本地调试通过 Vite 启动，默认端口 `5173` |
| 应用服务端 | Spring Boot 后端服务 | 本地启动后默认端口 `8080` |
| 数据库服务器 | MySQL 8 实例 | 默认数据库名 `finance_management_system` |
| 本地文件目录 | `backend/backups/` 备份目录 | 保存数据库备份文件及恢复记录 |

## 4. 功能结构设计

### 4.1 功能模块划分

| 模块名称 | 主要功能 | 主要角色 |
| --- | --- | --- |
| 登录与权限管理 | 登录、退出、角色识别、路由守卫 | 管理员、财务员 |
| 用户管理 | 用户新增、修改、启停、角色分配 | 管理员 |
| 部门管理 | 部门基础信息维护 | 管理员 |
| 账户管理 | 账户基础信息维护 | 管理员 |
| 类别管理 | 收入类别、支出类别维护 | 管理员 |
| 收入记录管理 | 收入记录新增、修改、查询、状态维护 | 管理员、财务员 |
| 支出记录管理 | 支出记录新增、修改、查询、状态维护 | 管理员、财务员 |
| 报表统计 | 仪表盘、周报、月报、年报、自定义区间报表 | 管理员、财务员 |
| 备份管理 | 备份创建、下载、恢复、历史追踪 | 管理员 |

### 4.2 角色权限设计

| 功能项 | 管理员 ADMIN | 财务员 FINANCE |
| --- | --- | --- |
| 登录与退出 | 支持 | 支持 |
| 首页仪表盘 | 支持 | 支持 |
| 用户管理 | 支持 | 不支持 |
| 部门、账户、类别管理 | 支持 | 不支持 |
| 收入记录、支出记录 | 支持 | 支持 |
| 报表中心 | 支持 | 支持 |
| 备份管理 | 支持 | 不支持 |

### 4.3 主要业务流程

1. 登录流程：用户输入账号密码，后端校验账号状态和密码，返回令牌及角色集合，前端据此加载菜单。
2. 主数据维护流程：管理员维护部门、账户和类别等基础资料，供收支业务引用。
3. 收支录入流程：业务人员录入金额、部门、账户、类别、发生日期和备注，系统校验数据有效性后生成业务编号并保存。
4. 报表统计流程：用户选择统计维度与时间范围，系统聚合收入和支出数据，返回汇总值、明细列表与趋势数据。
5. 备份恢复流程：管理员创建数据库备份，系统记录文件路径与时间；恢复时再次选择备份记录并执行恢复。

## 5. 数据设计概要

### 5.1 核心数据对象

| 数据对象 | 作用 | 关键字段 |
| --- | --- | --- |
| `sys_user` | 存储系统用户信息 | `username`、`password`、`real_name`、`status` |
| `sys_role` | 定义角色 | `role_code`、`role_name` |
| `sys_user_role` | 维护用户与角色关系 | `user_id`、`role_id` |
| `dept_info` | 部门基础信息 | `dept_code`、`dept_name`、`status` |
| `account_info` | 财务账户信息 | `account_code`、`account_name`、`status` |
| `income_category` | 收入类别主数据 | `category_code`、`category_name`、`status` |
| `expense_category` | 支出类别主数据 | `category_code`、`category_name`、`status` |
| `income_record` | 收入业务记录 | `record_no`、`amount`、`occurred_on`、`status` |
| `expense_record` | 支出业务记录 | `record_no`、`amount`、`occurred_on`、`status` |
| `backup_record` | 备份历史记录 | `backup_name`、`file_path`、`created_at`、`restored_at` |

### 5.2 数据关系设计

- 用户与角色是多对多关系，通过 `sys_user_role` 实现关联。
- 收入记录和支出记录都必须关联部门、账户和类别。
- 备份记录与业务数据解耦，单独承担备份历史追踪职责。

### 5.3 数据约束

- 主数据统一使用 `status` 表示启用或停用，停用数据不允许被新增业务继续引用。
- 收支金额必须大于 0。
- 收入记录编号采用 `IN + yyyyMMdd + 三位流水号`。
- 支出记录编号采用 `EX + yyyyMMdd + 三位流水号`。

## 6. 接口设计概要

### 6.1 接口分组

| 接口前缀 | 说明 |
| --- | --- |
| `/auth` | 登录与退出 |
| `/users` | 用户管理 |
| `/departments` | 部门管理 |
| `/accounts` | 账户管理 |
| `/income-categories` | 收入类别管理 |
| `/expense-categories` | 支出类别管理 |
| `/incomes` | 收入记录管理 |
| `/expenses` | 支出记录管理 |
| `/reports` | 仪表盘和统计报表 |
| `/backups` | 备份记录、下载与恢复 |

### 6.2 统一响应规范

系统接口统一返回 `ApiResponse(code, message, data)` 结构，其中 `code` 表示处理状态，`message` 表示结果说明，`data` 承载业务数据，前端基于该结构做统一提示和错误处理。

## 7. 安全与质量属性设计

### 7.1 安全设计

- 前端使用路由守卫检查登录状态和角色权限。
- 后端在登录时校验账号状态和密码合法性。
- 业务接口对非法参数、数据不存在和状态不合法场景统一抛出异常。

### 7.2 性能设计

- 主数据模块优先采用简单列表查询，保持实现稳定。
- 报表模块使用聚合 SQL 直接统计，减少额外对象转换。
- 演示数据规模以课程设计场景为准，不引入复杂缓存与分布式组件。

### 7.3 可维护性设计

- 代码按 `controller`、`service`、`mapper`、`entity`、`dto`、`vo` 分层组织。
- 文档、源码、数据库、建模图按目录独立归档。
- 通过统一命名降低课程答辩和后续维护的理解成本。

## 8. 运行与部署设计

### 8.1 运行环境

| 项目 | 建议环境 |
| --- | --- |
| 操作系统 | Windows、macOS 或 Linux |
| JDK | 17 |
| Node.js | 18 及以上 |
| Maven | 3.9 及以上 |
| 数据库 | MySQL 8 |
| 浏览器 | Chrome 或 Edge 最新稳定版 |

### 8.2 启动方式

1. 执行数据库脚本 `11-数据库/财务管理系统.sql` 初始化数据库。
2. 进入 `10-源代码/backend` 执行 `mvn spring-boot:run`。
3. 进入 `10-源代码/frontend` 执行 `npm install` 和 `npm run dev`。
4. 浏览器访问前端地址，使用演示账号完成业务操作。
