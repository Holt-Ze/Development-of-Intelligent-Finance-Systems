## 4. 提交前环境检查

### 4.1 代码运行环境

| 项目 | 建议版本 |
| --- | --- |
| JDK | 17 |
| Maven | 3.9 及以上 |
| Node.js | 18 及以上 |
| npm | 与 Node.js 配套版本 |
| MySQL | 8 |

### 4.2 构建验证命令

1. 后端进入 `10-源代码/backend` 执行 `mvn test`。
2. 前端进入 `10-源代码/frontend` 先执行 `npm install`，再执行 `npm run build`。
3. 数据库执行 `11-数据库/财务管理系统.sql`，确认核心表可以成功创建。

## 5. 目录与命名规范

- 目录统一采用“编号 + 中文名称”的形式。
- Word 文档统一命名为“文档名称-财务管理系统.docx”。
- Markdown 源文档与 Word 文档保持同名，便于后续继续修改。
- 截图和录屏文件应体现功能主题，避免使用无意义文件名。

## 6. 提交步骤

### 6.1 文档核对

1. 确认 10 类核心文档全部存在且可打开。
2. 检查文档封面、标题、章节编号、页码和文件名是否统一。
3. 检查需求、设计、测试、用户手册之间的功能名称是否一致。

### 6.2 代码核对

1. 确认前后端源码完整，无缺失关键配置文件。
2. 确认 `package-lock.json`、`pom.xml` 和数据库脚本已纳入提交包。
3. 清理无关临时文件，但保留课程演示所需的配置与样例数据。

### 6.3 打包提交

1. 以项目根目录为单位进行压缩归档。
2. 压缩包建议命名为 `财务管理系统课程设计-小组名-日期.zip`。
3. 若教师有平台命名要求，以教师要求为准。

## 8. 常见问题

### 8.1 前端构建失败

若执行 `npm run build` 提示 `vite: command not found`，说明前端依赖尚未安装，应先执行 `npm install`。

### 8.2 前端构建出现大体积 chunk 警告

若构建输出提示某些 chunk 超过 `500 kB`，说明前端打包体积偏大，但不影响当前课程设计版本提交。若后续继续优化，可考虑按页面拆包或配置 `manualChunks`。

### 8.3 数据库启动后页面无数据

应确认是否已执行初始化脚本，并检查后端配置中的数据库地址、账号和密码是否正确。

### 8.4 文档名称不统一

提交前统一检查文件命名，避免出现“说明书”“设计书”“报告书”混用导致教师难以检索。

## 9. 结论

通过本提交手册，可在最终归档前完成一次针对文档、代码、数据库与测试材料的整体核对，降低漏交风险，提高课程设计成果的规范性和可验收性。
