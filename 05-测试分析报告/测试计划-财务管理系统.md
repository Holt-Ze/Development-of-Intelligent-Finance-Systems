# 测试计划

## 1. 文档目的

本文档用于明确财务管理系统测试活动的目标、范围、策略、资源安排、进度计划与准入准出标准，作为测试执行、缺陷跟踪和测试总结的管理依据。

## 2. 测试依据

- 《需求规格说明书-财务管理系统》
- 《概要设计-财务管理系统》
- 《详细设计-财务管理系统》
- 当前前后端源码与数据库脚本
- 《测试用例-财务管理系统》

## 3. 测试目标

- 验证系统核心功能是否满足需求说明。
- 验证管理员与财务员两类角色的权限边界是否正确。
- 验证数据录入、统计查询、备份恢复等关键流程是否完整可用。
- 验证系统在课程设计演示环境下可以稳定启动、构建和展示。

## 4. 测试范围

### 4.1 范围内

- 登录与退出
- 用户管理
- 部门管理
- 账户管理
- 收入类别管理
- 支出类别管理
- 收入记录管理
- 支出记录管理
- 财务统计与报表
- 数据备份与恢复
- 前端构建与后端基础编译验证

### 4.2 范围外

- 高并发压测
- 大规模数据量专项测试
- 第三方系统集成测试
- 多浏览器兼容性深度验证
- 生产部署与运维监控测试

## 5. 测试策略

### 5.1 测试类型

| 测试类型 | 目标 | 说明 |
| --- | --- | --- |
| 功能测试 | 验证需求功能是否实现正确 | 以页面操作和接口响应为主 |
| 权限测试 | 验证角色访问边界 | 重点检查菜单、页面与接口入口 |
| 数据校验测试 | 验证金额、日期、状态及主数据引用规则 | 覆盖正常值、空值和非法值 |
| 报表测试 | 验证统计口径与筛选条件 | 重点覆盖周报、月报、年报、自定义区间 |
| 构建验证测试 | 验证工程可编译和可打包 | 使用 `mvn test` 与 `npm run build` |
| 回归测试 | 验证缺陷修复后主流程未受影响 | 聚焦登录、录入、查询、备份四条主链路 |

### 5.2 测试方法

- 手工黑盒测试为主，依据需求和页面行为设计用例。
- 使用等价类、边界值、场景法和权限矩阵法设计测试项。
- 对后端采用构建级验证，确认源码可编译、依赖完整。

## 9. 准入准出标准

### 9.1 测试准入标准

- 需求、设计和页面清单已经基本稳定。
- 数据库脚本可以完成初始化。
- 前后端代码具备可启动或可构建的基本条件。
- 核心测试用例已经整理完成。

### 9.2 测试准出标准

- 核心功能用例执行完成率达到 100%。
- 严重和高优先级缺陷关闭或有明确规避方案。
- 主流程“登录 -> 录入 -> 报表 -> 备份”可以连续演示。
- 主要文档、日志和截图已经归档。

## 10. 缺陷管理规则

| 优先级 | 定义 | 处理要求 |
| --- | --- | --- |
| P1 | 阻断主流程，无法演示 | 必须优先修复并回归 |
| P2 | 功能异常但有替代路径 | 在结项前修复 |
| P3 | 文案、样式或轻微体验问题 | 可集中修订 |
| P4 | 建议优化项 | 记录到后续改进清单 |

## 11. 风险与应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 前端依赖未安装 | 无法执行构建或启动 | 提前执行 `npm install` 并记录依赖版本 |
| 数据初始化不完整 | 页面无数据或报表为空 | 统一使用 SQL 初始化脚本 |
| 权限口径理解不一致 | 测试结果反复变更 | 先以需求文档中的权限矩阵为准 |
| 文档与代码脱节 | 答辩时出现描述不一致 | 测试后同步更新文档与日志 |

## 13. 结论

本测试计划以课程设计演示成功为目标，采用“小范围、重主流程、重可追踪”的策略，确保系统在有限周期内达到可验证、可归档和可答辩的交付标准。
